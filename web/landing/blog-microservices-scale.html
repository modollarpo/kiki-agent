Scaling to 10M Messages/Hour: Our Go Microservices Architecture

January 10, 2026 • 8 min read • By James Doe, Principal Engineer

Inside the SyncEngage pipeline: how we move 10M retention messages per hour with Go, Kafka, Kubernetes, and predictable idempotent workflows.

← Back to Blog

Key Outcomes

10M/hr

Sustained delivery throughput with p95 latency < 450ms.

5x cost down

Kafka tiered storage + protobuf reduced infra spend vs. JSON + REST.

99.98% success

Exactly-once semantics via idempotent keys and outbox patterns.

HPA-ready

Horizontal Pod Autoscaler driven by queue depth and CPU.

System Diagram

Events flow from campaign triggers into Kafka topics, processed by Go workers. Each worker handles batching, template rendering, channel selection (email/SMS/push), and downstream provider API calls. We use gRPC between services and HTTP only at the edge providers.

[Triggers] -> [Kafka topics] -> [Go Workers] -> [Provider APIs]
                 |                  |                 |
             Schema Reg        gRPC services      DLQ + Replay
                 |                  |                 |
            Observability     Idempotent keys    S3 archival

Idempotency & Ordering

  • Kafka keys set to user_id + campaign_id + variant_id for deterministic partitioning.
  • Outbox pattern from Postgres: write once, deliver-many with idempotent provider tokens.
  • Dedup cache (Redis) with 24h TTL to prevent double-delivery during retries.
  • Dead-letter queues with replay UI and alerting into PagerDuty.

Performance Tuning

  • Go workers with worker pools sized by CPU cores; backpressure via Kafka consumer lag.
  • Protobuf payloads and gRPC streaming cut network overhead by 38% vs. REST+JSON.
  • Connection pooling to downstream providers; exponential backoff with jitter.
  • Configurable batch sizes per channel (email: 200, SMS: 50, push: 500).
  • Autoscaling triggers on consumer lag and p95 latency, not only CPU.

Reliability

  • Canary topics per provider to detect SLA regression before global rollout.
  • Chaos testing on Kafka partitions and provider 429/5xx responses.
  • Health checks: gRPC health protocol + synthetic messages per minute.
  • Tracing: OpenTelemetry with spans covering fetch, render, send, and ack.

Results

SyncEngage now handles 10M messages/hour with 99.98% delivery success, p95 latency <450ms, and 5x lower infra cost. Engineering teams ship new channels safely using canary topics and idempotent workflows.
